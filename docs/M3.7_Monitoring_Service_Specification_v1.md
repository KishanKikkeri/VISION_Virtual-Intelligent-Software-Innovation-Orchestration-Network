Depends on: M2 (Product), M3.1 (Architecture), M3.2 (Repository), M3.3 (Engineering),

## Preface

M3.7 introduces the platform's first **continuous-running** department. Every
prior department (M2–M3.6) is invoked once per project phase, runs a bounded
LangGraph to completion, and terminates. Monitoring instead evaluates platform
health on a recurring cycle for as long as the process is alive.

This document freezes the decisions needed to build that without breaking the
conventions the other seven departments already rely on — in particular the
existing `infrastructure/monitoring/telemetry.py` module (Prometheus/OTel,
already shipped in Sprint 3) and the existing `AASC_EVENTS` JetStream stream
(`infrastructure/messaging/nats_client.py`), neither of which currently expose
a `monitoring.*` surface.

Reconnaissance performed before writing this spec (per the handover's
Recommended Execution Order, step 1):

* `core/runtime/factory.py` — 86 agents currently registered across 8
  departments (manager, product, devops, docs, architecture, engineering,
  qa, security). **No `monitoring_*` agent IDs are reserved.** The naming
  convention is `<noun>_head`, `<noun>_lead`, `<noun>_worker`, department
  set via the `department=` field on `AgentSpec`.
* `infrastructure/messaging/nats_client.py` — `STREAM_CONFIGS[0]` (AASC_EVENTS)
  lists subject prefixes explicitly: `manager.>`, `product.>`, `architecture.>`,
  `engineering.>`, `repository.>`, `qa.>`, `security.>`, `devops.>`, `docs.>`,
  `token_ledger.>`. **`monitoring.>` is absent** and must be added (Appendix A.1).
* `core/contracts/__init__.py` — `artifact_type` is a free-form `str` field
  (`ArtifactRef.artifact_type: str`), not a closed enum. New artifact type
  strings (`metrics_snapshot`, etc.) require **no contract change**, only
  registration through the existing Artifact Registry, exactly as DevOps did
  for `deployment_plan`.
* `infrastructure/monitoring/telemetry.py` already exposes Prometheus
  `Counter`/`Gauge`/`Histogram` series (`aasc_token_usage_total`,
  `aasc_workflow_phase_duration_seconds`, `aasc_agent_run_duration_seconds`,
  `aasc_active_projects_total`, `aasc_deployments_total`,
  `aasc_websocket_connections`, `aasc_nats_messages_published_total`, …) and a
  tracer. Monitoring Service **must consume this existing registry**, not
  stand up a second one.
* `infrastructure/websocket/manager.py` — `WebSocketManager.broadcast(project_id, …)`
  is project-scoped; `broadcast_system(event_type, payload)` is platform-wide.
  Monitoring is platform-wide by nature and must default to `broadcast_system`.
* No service in the codebase currently runs a background scheduling loop —
  every `main.py` lifespan only wires DB/NATS/telemetry once at startup. The
  continuous-loop mechanism is therefore a genuinely new decision, frozen
  below (§7), not something to copy from an existing service.
* `docker/grafana/provisioning/` exists but is currently empty — reserved by
  infra scaffolding for this milestone.

## 0. Frozen cross-service decisions

### Decision 1 — Continuous execution is bounded-cycle, not a literal infinite graph

A single never-terminating LangGraph invocation cannot be checkpointed,
resumed, or unit-tested the way every other W-* graph in this platform is
(`services/qa/workflows/qa_graph.py`, `services/security/workflows/security_graph.py`,
`services/devops/workflows/devops_graph.py` all rely on discrete Postgres
checkpoints per run).

Monitoring's graph (`W-MONITORING`) is therefore a **normal bounded graph**
(`Collect → Aggregate → Analyze → Health Score → {Dashboard | Alert} →
Incident? → Publish → END`) that a lightweight scheduler re-invokes on a
fixed interval, using the **same `thread_id`** across cycles so the
LangGraph checkpointer preserves rolling state (moving averages, alert
dedup windows, consecutive-breach counters) between runs. "Continuous" means
*the scheduler never stops issuing new cycles while the process is alive*,
not that any single graph execution is unbounded. This is enforced in
`services/monitoring/main.py`'s FastAPI `lifespan`, via an `asyncio` background
task — not inside a graph node.

Default cycle interval: **30 seconds**, configurable via
`MONITORING_CYCLE_INTERVAL_SECONDS` in `core/config/settings.py`.

### Decision 2 — Metrics providers wrap existing telemetry; they do not duplicate it

For platform-internal series (LLM tokens/cost, workflow phase duration, agent
run duration, active projects, deployments, WebSocket connections, NATS
throughput) the **Infrastructure Metrics Worker** and **Application Metrics
Worker** read from the Prometheus client registry already populated by
`infrastructure/monitoring/telemetry.py` (in-process `REGISTRY.collect()`,
no second scrape target). For externally-owned systems with no existing
telemetry hook (PostgreSQL connection/replication stats, Qdrant collection
stats, NATS JetStream stream/consumer stats, Docker container stats), each
gets its own thin provider under `services/monitoring/providers/` following
the Provider Pattern used everywhere else in the codebase (one class per
external system, `async def collect() -> list[MetricSample]`).

### Decision 3 — Health score is a deterministic weighted composite, frozen here

```
component_score(c) ∈ [0, 100]   # per monitored component, see §5
health_score = Σ (component_score(c) * weight(c)) / Σ weight(c)

Default weights:
  postgres=3, nats=3, qdrant=2, docker=2, websocket=1,
  llm_providers=3, agent_runtime=3, repository=2, deployments=2

Thresholds (frozen):
  HEALTHY   : health_score >= 90
  WARNING   : 70 <= health_score < 90
  CRITICAL  : health_score < 70
```

Component scores and weights are configuration, not hardcoded per-agent
judgment calls — this keeps the score reproducible in tests (§11).

### Decision 4 — Dashboards are DB-backed; Grafana provisioning is an export target, not the source of truth

`dashboards` / `dashboard_widgets` tables (owned by Monitoring, §9) are
authoritative. `GET /dashboard` is served from Postgres so the frontend never
depends on Grafana being reachable. The **Dashboard Worker** additionally
renders the same configuration to Grafana-provisioning JSON and writes it to
`docker/grafana/provisioning/dashboards/` for operators who want the Grafana
UI — this is a one-way export, never read back.

### Decision 5 — Incident escalation threshold, frozen for M3.8 handoff

An `incident_candidate` artifact is published (and `monitoring.incident`
emitted) when either:
* `health_score` stays `CRITICAL` for **3 consecutive cycles** (default,
  `MONITORING_INCIDENT_BREACH_CYCLES`), or
* a hard-failure signal arrives directly (e.g. a monitored component is
  unreachable, not merely degraded).

Anything less persistent only produces `monitoring.warning` / a `dashboard`
update. M3.8 (Incident Response Service) consumes `monitoring.incident` and
`incident_candidate` — Monitoring never resolves incidents itself.

## 2. Team structure

* **Metrics Lead** owns *collection* — no analysis, no judgment calls, purely
  gathering `MetricSample` rows via providers and writing to `metric_samples`.
* **Observability Lead** owns *interpretation of raw signal* — logs and
  traces, feeding derived signals (error rate, p95 latency) into Aggregate.
* **Alerting Lead** owns *communication* — the only team allowed to publish
  `monitoring.alert` / `monitoring.warning` / `monitoring.incident` and to
  write `alerts` / `alert_history`. Metrics and Observability teams never
  publish alerts directly, mirroring the separation-of-duties precedent set
  in M3.3 (Decision 3, `docs/M3.3_Engineering_Service_Specification_v1.md`).

---

## 3. Task decomposition (one cycle)

1. **Collect** — Metrics Lead runs both workers concurrently (`asyncio.gather`);
   each provider returns `List[MetricSample]`; failures are per-provider
   (one dead component degrades that component's score, never aborts the cycle).
2. **Aggregate** — samples + log/trace derived signals are rolled into one
   `component_score` per monitored system (§0 Decision 3).
3. **Analyze** — capacity trend (linear slope over trailing window) and
   performance deltas vs. previous cycle are computed; feeds `capacity_forecast`.
4. **Health Score** — weighted composite (§0 Decision 3) → `HEALTHY` /
   `WARNING` / `CRITICAL`.
5. **Dashboard / Alert** — both always run (Dashboard always refreshes;
   Alert only emits on threshold crossing or new breach, not every cycle,
   to avoid alert fatigue).
6. **Incident?** — breach-streak counter check (§0 Decision 5).
7. **Publish** — `monitoring.metrics.updated` always; `monitoring.phase.completed`
   always; WebSocket `broadcast_system` always; `monitoring.incident` conditionally.

## 5. Health scoring & alerting standards

Component score inputs (frozen per component so tests are deterministic):

| component        | healthy signal                          | degraded signal                         |
|-------------------|-------------------------------------------|--------------------------------------------|
| postgres           | connection pool utilization, replication lag | pool exhaustion, lag > threshold            |
| nats                | JetStream consumer lag, ack pending count    | consumer stalled, DLQ growth                |
| qdrant              | query latency, collection availability       | latency spike, collection unreachable       |
| docker              | container running/healthy count               | container restarts, OOM kills               |
| websocket           | `aasc_websocket_connections` gauge delta       | connection churn spike                      |
| llm_providers       | `aasc_token_usage_total` / error ratio         | provider error rate, rate-limit responses   |
| agent_runtime       | `aasc_agent_run_duration_seconds` p95           | timeout rate, escalation rate                |
| repository          | webhook delivery success rate                   | webhook failures, PR stuck > SLA             |
| deployments         | `aasc_deployments_total{status=healthy}` ratio  | rollback rate                               |

Alert severities (new `AlertSeverity(str, Enum)` in `services/monitoring/models.py`,
following the `DeploymentStatus` / `DevOpsTaskStatus` enum pattern already used
in `services/devops/models.py`): `INFO`, `WARNING`, `CRITICAL`.

Alert dedup: identical `(component, severity)` pair suppressed for
`MONITORING_ALERT_DEDUP_SECONDS` (default 300s) to avoid re-alerting every
30-second cycle.

## 6. Data collection & provider rules

* Providers live in `services/monitoring/providers/`, one file per system:
  `postgres_provider.py`, `qdrant_provider.py`, `nats_provider.py`,
  `docker_provider.py`, `telemetry_provider.py` (wraps existing Prometheus
  registry), `repository_provider.py`, `deployment_provider.py`.
* Common interface:
  ```python
  class MetricsProvider(Protocol):
      async def collect(self) -> list[MetricSample]: ...
  ```
* Monitoring **never** writes to another department's tables (`deployments`,
  `repositories`, `pull_requests`, etc.) — it reads them read-only, same rule
  the handover states for source code ("Monitoring must never write source
  code" extends here to "Monitoring must never write another department's
  operational tables").
* Docker collection uses the Docker Engine API (read-only `stats`/`ps`),
  never `docker exec` / build / run.

## 7. Continuous execution model

See §0 Decision 1 for the rationale. Mechanics:

```python
# services/monitoring/main.py (lifespan, sketch)
async def _monitoring_loop(app_state):
    thread_id = "monitoring-continuous"   # stable across cycles
    while not app_state.shutting_down:
        await run_monitoring_cycle(thread_id=thread_id)
        await asyncio.sleep(settings.monitoring_cycle_interval_seconds)
```

The loop is started as an `asyncio` background task in the FastAPI
`lifespan` (same lifespan shape as `services/devops/main.py`) and cancelled
on shutdown — no `while True` inside any LangGraph node itself.

## 8. LangGraph workflow (W-MONITORING)

```
collect_metrics
      │
      ▼
  aggregate
      │
      ▼
   analyze
      │
      ▼
 health_score
      │
 ┌────┴─────┐
 ▼          ▼
dashboard  alert     (both always execute — dashboard always refreshes,
 │          │         alert conditionally emits per §5 dedup rule)
 └────┬─────┘
      ▼
  incident?
      │
 no ──┴── yes
      │        │
      ▼        ▼
   publish   incident_response_handoff → publish
      │
      ▼
     END          (of THIS cycle — scheduler re-invokes per §7)
```

State (`MonitoringState`, `TypedDict`) persists across cycles via the
Postgres checkpointer keyed on the stable `thread_id`, carrying forward:
`consecutive_critical_count`, `last_alert_at: Dict[str, datetime]` (for
dedup), and trailing sample windows for `capacity_forecast`. Routing
functions follow the existing `route_after_*` naming convention
(`services/monitoring/routing.py`), mirroring `services/devops/routing.py`.

Retry/DLQ: a failed provider inside `collect_metrics` degrades that
component's score to 0 for the cycle rather than retrying/escalating —
continuous monitoring must never block on a single flaky collector.

```

           monitoring.warning, monitoring.incident,
```

## 13. Monitoring Coding Contract

* Pydantic models for all in-memory/API shapes (`services/monitoring/models.py`,
  `schemas.py`) — SQLAlchemy ORM only in `infrastructure/database/models.py`.
* Provider Pattern for every external system (§6); Repository Pattern for
  DB access (`services/monitoring/integration/`).
* No hidden globals — cycle state lives in `MonitoringState`, not module-level
  mutable variables (the `thread_id`-keyed checkpointer is the only persistence
  mechanism across cycles).
* Never modify `AgentFactory`, `BaseAgent`, or any other department's models.
* Never write to `deployments`, `repositories`, `pull_requests`, or any table
  owned by another department — read-only access only.

## Addendum (post-freeze, discovered during implementation) — Decision 6

The existing Artifact Registry's `artifacts` table has a `NOT NULL` FK
`project_id → projects.id`, and `projects.owner_id` is itself `NOT NULL` FK
`→ users.id`. Every prior department is invoked *within* a single project's
workflow, so this was never exercised as a gap before. Monitoring is the
platform's first cross-project department and has no single project to
anchor to.

**Resolution:** `services/monitoring/integration/platform_anchor.py`
idempotently creates one sentinel system user + one sentinel
`"Platform Monitoring"` project at startup; every Monitoring
`TaskInput`/`AgentContext`/artifact anchors to that project_id. One additive
row in two already-existing tables — no schema change, no modification to
`AgentFactory`/`BaseAgent`/any other department. See
`docs/M3.7_Monitoring_Service_Handover.md` for the full writeup and the note
to M3.8 about reusing this same anchor rather than creating a second one.
