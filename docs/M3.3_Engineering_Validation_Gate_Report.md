## 4. Gaps found and fixed

### 4.1 Missing trigger: Architecture approval never started Engineering
`services/manager/main.py`'s `/projects/{id}/approve` endpoint had:
```python
if artifact_type == "requirements":
    await _run_department_pipeline(..., "run_architecture_pipeline", ...)
```
...and nothing for `artifact_type == "architecture"`. Engineering was fully
built and registered but **nothing in the live system would ever call it**.
Added the missing `elif` branch. This is the one change that makes your
"Integration" checklist section executable at all.

### 4.2 Wrong artifacts fetched for every non-Product department
`_run_department_pipeline()` hardcoded the Product artifact list
(`feature_spec_doc`, `requirements_doc`, `user_stories_doc`,
`acceptance_criteria`) for **every** department, including Engineering. It
happened to be harmless for Architecture (which also reads Product's
output), but would have handed Engineering an empty context — no
`architecture_blueprint`, `openapi_spec`, `database_schema`, or
`ui_blueprint` — meaning `EngineeringHead` would have run its whole pipeline
against nothing. Replaced the hardcoded list with a per-department map.

### 4.3 Docker Compose had no Manager or Repository container
Only the Phase-1 `demo` app was in `docker/docker-compose.yml`. Added
`manager` (port 8001) and `repository` (port 8006) service blocks so
`docker compose up` actually stands up the services this checklist needs.
(Caught and fixed a YAML syntax error in my own edit — missing space after
a `GITHUB_DEFAULT_OWNER:` key — by actually parsing the file with
`yaml.safe_load()`, not just reading it.)

## 5. Gaps found and **not** fixed (flagged for your call)

- **NATS-driven phase advancement is asymmetric.** Manager subscribes to
  `product.requirements.completed` to flip the lifecycle graph's internal
  `phase_status`, but there's no equivalent subscription for
  `architecture.design.completed` or `engineering.phase.completed`. The
  *synchronous* `_run_department_pipeline()` call still works regardless
  (that's the actual execution path), but the LangGraph `LifecycleState`
  itself won't reflect Architecture/Engineering completion the same way it
  reflects Requirements completion. Pre-existing asymmetry from before
  M3.3, not caused by it, but worth closing before you rely on the
  lifecycle graph's own state for anything user-facing.
- **`feature_name` defaults to `"default"`.** Manager doesn't currently
  pass a real feature name into Engineering's context, so
  `integration/default` branches would be created regardless of what the
  project actually is. Cosmetic today (single-feature-per-project), but
  will need a real value once multiple features run per project.
- **Engineering doesn't write its own `AuditRepository` rows** — see the
  audit events row in §3.

None of these block Engineering's *own* correctness — they're seams in how
Manager drives every department, not defects in the Engineering pipeline
itself — but I'm flagging them rather than letting "Engineering is closed"
imply the whole chain is production-hardened end to end.
