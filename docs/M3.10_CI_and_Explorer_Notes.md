# M3.10 Follow-up — CI/CD Quality Gates & Developer Experience

Builds on M3.10 (Workflow Refinement). No workflow business logic changed in
this pass — this is tooling and developer-experience work only.

---

## CI/CD & Quality Gates

### 1. Workflow validation in CI

`scripts/ci/validate_workflows.py` — a standalone CLI gate (no pytest
required to run it, though it reuses the same modules the M3.10 test suite
does):

```
python scripts/ci/validate_workflows.py               # health + lint
python scripts/ci/validate_workflows.py --health-only
python scripts/ci/validate_workflows.py --lint-only
python scripts/ci/validate_workflows.py --fail-on-warnings
```

Exit 0 = all 10 workflows healthy and lint-clean (against baseline — see
below). Exit 1 = a real failure. Exit 2 = the script itself couldn't even
import the validator/linter (distinguished from "the platform is broken" so
a CI dashboard can tell "we broke the checker" from "we broke a workflow"
at a glance).

Wired into `.github/workflows/workflow-validation.yml`'s `validate-and-lint`
job, which also runs the 393-test `test_m310_workflows.py` suite and
uploads the validation report as a build artifact.

### 2. Auto-generate Mermaid docs during builds

`scripts/ci/generate_docs.py` wraps `graph_exporter.write_mermaid_docs()` +
`workflow_docs.write_all_docs()` (the same pair used to produce
`docs/workflows/` in M3.10):

```
python scripts/ci/generate_docs.py                 # regenerate in place
python scripts/ci/generate_docs.py --check          # drift check, no writes
```

`--check` generates fresh docs into a scratch temp dir and diffs them
file-by-file against the committed `docs/workflows/` — it fails (exit 1,
with a unified diff) if a workflow graph changed but the docs weren't
regenerated and committed alongside it. On `pull_request`, CI runs
`--check`; on `push` to `main`, CI regenerates and auto-commits via
`stefanzweifel/git-auto-commit-action` (the `docs` job in the same
workflow file), and uploads `docs/workflows/` as a build artifact either
way.

### 3. Graph linting & architecture checks

New: `services/integration/diagnostics/graph_linter.py`. Deliberately
separate from `workflow_validator.WorkflowReport.healthy` (runtime
correctness — does it build, is everything reachable): this is
*architecture/style* — does the graph follow the platform's conventions.

| Rule | Severity | Checks |
|---|---|---|
| LINT-E001 | error | workflow builder failed to import/build |
| LINT-E002 | error | no entry node from `__start__` |
| LINT-E003 | error | graph has no `__end__` node |
| LINT-E004 | error | `__end__` exists but isn't reachable |
| LINT-E005 | error | node unreachable from `__start__` |
| LINT-E006 | error | conditional route targets an unknown node |
| **LINT-E007** | **error** | **a `route_*` function is imported into a graph module but never passed to `add_conditional_edges()`** |
| LINT-W001 | warning | isolated node (no edges at all) |
| LINT-W002 | warning | structural dead end (no outgoing route) |
| LINT-W003 | warning | routing function doesn't follow `route_*` naming |
| LINT-W004 | warning | node name isn't snake_case |
| LINT-W005 | warning | graph depth exceeds 25 stages (consider decomposing) |

**LINT-E007 is the interesting one**: it's a direct, automated version of
the investigation that found QA's unreachable `dlq` node in M3.10 — "a
routing function was written and imported but never wired in." Running it
against the current platform immediately found **3 more instances** beyond
the one already fixed: `engineering.route_after_aggregate`,
`security.route_after_fan_out`, `qa.route_after_coverage`, and
`qa.route_after_execute`. These were **not** in the M3.10 spec's list of
known genuine issues, so — consistent with M3.10's own instruction not to
"blindly wire" a routing function without first confirming intended design
— they were **not** investigated or fixed here. Turning on a new lint rule
against an established codebase routinely surfaces this kind of debt; the
right response is a **baseline**
(`services/integration/diagnostics/lint_baseline.json`), not silently
disabling the rule or failing every future build until someone does that
investigation. `scripts/ci/validate_workflows.py` passes today because
these 4 are recorded in the baseline; any *new*, non-baselined LINT-E007
(or any other rule) finding fails the build immediately. Removing a
workflow from the baseline once it's actually wired or cleaned up is a
one-line JSON edit.

Tests: `tests/foundation/test_m310_ci_quality_gates.py` (24 tests) —
synthetic-graph tests pinning down each rule in isolation, real-platform
lint-pass tests, baseline-comparison unit tests, and subprocess-level tests
that actually invoke both CI scripts and check their exit codes/output.

## Developer Experience

### 4. Web-based workflow explorer

New: `services/integration/static/explorer/index.html` — a single static
page (no build step; vanilla JS + Mermaid.js via CDN), served at
`GET /explorer/` (mounted in `main.py` via `StaticFiles`). It's a pure
client for the `/platform/*` API — no new backend logic beyond the traces
endpoint below.

- **Workflows tab**: sidebar lists all 10 workflows with a live health dot;
  a search box filters by workflow name *or* node name (typing a node name
  like `dlq` jumps straight to the workflow that contains it). Selecting a
  workflow renders its live Mermaid diagram (fetched from
  `/platform/workflows/mermaid/{name}`) plus a detail panel — stats,
  routing table, parallel branches, warnings/errors — from
  `/platform/workflows/{name}`, with a link out to the full
  `docs/workflows/{name}.md` page.
- **Execution Traces tab**: enter a `project_id`, see its real audit-event
  timeline (event type, actor, entity, payload, timestamp) rendered
  chronologically. Backed by a new endpoint, `GET
  /platform/traces/{project_id}`, which reads the actual `AuditEvent`
  append-only table (`infrastructure/database/models.py` calls it "the
  black box recorder") via `AuditRepository.list_for_project` — not
  synthetic/mocked data. A project with no activity yet gets an empty list
  back, not a 404; a process with no DB configured (e.g. a bare test
  client) gets an explicit `note` field rather than a crash.
- **Architecture tab**: see below.

Design note: dark, information-dense "control room" theme (deep charcoal
background, muted slate panels, teal/amber/coral status colors, monospace
for all graph/stat data) — deliberately chosen for an internal ops/
diagnostics tool rather than the cream-and-terracotta or near-black-with-
one-accent looks that read as generic AI-generated defaults.

### 5. Architecture dashboards and dependency maps

The explorer's **Architecture tab** (`loadArchitecture()` in the same
static page) renders, live from existing endpoints:

- **Platform readiness** gauges — one bar per category from
  `GET /platform/readiness` (registry, workflows, events, etc.), colored by
  score (green ≥90, amber ≥70, red below).
- **Department dependency map** — `GET /platform/dependencies`'s
  `phase_tiers` rendered as ordered rows of department chips (tier 1 →
  tier N), with orphan departments (no upstream dependency — `manager`,
  `docs`) called out distinctly, and a cycle warning if
  `dependency_graph.has_cycle()` ever returns true.
- **Agent registry** — total/expected agent counts and the department list
  from `GET /platform/registry`.

No new backend endpoints were needed for this tab — it's a pure
visualization layer over data the M3.9 platform-integration APIs already
expose.

Tests: appended to `tests/foundation/test_m310_workflows.py` — `/platform/
traces/*` endpoint tests, and static-mount tests confirming `/explorer/`
serves the page and `/docs/workflows/{name}.md` is reachable for all 10
workflows through the same FastAPI app.

## Known limitations / follow-ups

- The 4 baselined LINT-E007 findings are tracked, not fixed — see the
  baseline file's own comment for the investigation each would need before
  wiring or removing (same rigor as the QA `dlq` fix).
- The explorer's Mermaid rendering and dependency map are read-only
  snapshots on page load / workflow selection — there's no live
  auto-refresh or websocket push yet if a workflow definition changes while
  the page is open.
- `GET /platform/traces/{project_id}` surfaces whatever's in `AuditEvent`
  for that project; it doesn't yet correlate audit rows back to *which
  LangGraph node* was executing when each event was recorded, since
  `AuditEvent` doesn't currently carry a node-name field. Adding one would
  let the explorer draw traces directly onto the Mermaid diagram (e.g.
  highlighting the path a specific run actually took) — a natural next
  step, not implemented here.
- The GitHub Actions workflow is a complete, ready-to-use definition but
  wasn't executed against a real GitHub Actions runner as part of this
  work (no CI environment available here) — it was validated by running
  the equivalent commands (`pytest`, `scripts/ci/validate_workflows.py`,
  `scripts/ci/generate_docs.py --check`) directly and confirming their exit
  codes and output match what the workflow file expects.
