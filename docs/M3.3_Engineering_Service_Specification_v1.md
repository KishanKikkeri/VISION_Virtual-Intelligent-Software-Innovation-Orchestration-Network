## 0. Frozen cross-service decisions

These four decisions are locked and referenced throughout the rest of this spec.

### Decision 1 — `ui_blueprint` is an Architecture artifact

Frontend workers must never infer UI structure from `openapi_spec` alone — that makes the
frontend an LLM interpretation instead of an engineered system, and breaks traceability.
Architecture Service produces `ui_blueprint` (page hierarchy, navigation tree, routes,
component hierarchy, forms, tables, dialogs, user flows, state boundaries, API bindings,
permission mapping, responsive/accessibility requirements, design-system references) as a
**10th Architecture artifact**, subject to the same user-approval gate as the other 9.
This requires a small, additive amendment to the already-shipped Architecture Service —
see **Appendix A.1**. No frontend worker invents structure; every one of them reads
`ui_blueprint` as a required input (§4).

### Decision 2 — Hierarchical pull request model

Neither one-PR-per-task (too much review noise) nor one-PR-per-phase (unreviewable,
unrollback-able) is acceptable. Locked model:

```
Worker            → owns a small feature branch, one per task
                     (feature/<task-id>-<slug>, per M3.2 naming policy — unchanged)
Lead               → aggregates its team's related feature branches into ONE
                     integration branch, then opens ONE integration PR to develop
Department (Head)  → gives the department-level sign-off (Decision 3, Level 2)
                     that actually allows Repository Service to merge it
```

Example: `feature/TASK-101-jwt`, `feature/TASK-102-login`,
`feature/TASK-103-refresh` → `integration/backend_lead-authentication` → one PR into
`develop`.

**How aggregation happens without violating M3.2's "no merge commits" rule:** a Lead does
not `git merge` worker branches together. It takes the final file state each worker
produced (already validated by Code Review, §7) and calls Repository Service's
`commit_files` against the integration branch directly — one commit per aggregated
worker output, still squash-mergeable, still zero merge commits anywhere in the history.
Worker feature branches are retired (marked `merged` in Repository Service's `branches`
table, not deleted until the integration PR itself merges) once their content has been
folded into the integration branch.

This requires one small, additive Repository Service amendment — a 4th branch type,
`integration`. See **Appendix A.2**.

### Decision 3 — Three-level approval, enforced separation of duties

Repository Service's `pull_requests.status` stays exactly as built in M3.2
(`open → approved → merged`) — **no schema change to Repository Service**. The three
levels are an Engineering-owned governance concept layered on top of that single
Repository Service `approve` call:

| Level | Performed by | Purpose | Status label (Engineering-owned, not a Repository Service enum) |
|---|---|---|---|
| 1 — Automated Review | `code_review_lead` | coding standards, static analysis, architecture compliance, build verification (§7) | `AI_APPROVED` |
| 2 — Department Approval | `engineering_head` | feature completeness, cross-team integration, release readiness | `DEPARTMENT_APPROVED` |
| 3 — Human Approval (optional) | a human, via `manager_agent`'s existing user-approval flow | protected branches, production deployments, enterprise policy | `HUMAN_APPROVED` |

Mechanically:
- Level 1 is recorded as an Engineering-side field on the task/module record (§4's
  `CodeModule`, extended with `review_status`) — it does **not** call Repository Service.
- Level 2 is what actually triggers Repository Service's single
  `POST /pull-requests/approve` call, with `approved_by="engineering_head"`. This is the
  real merge-eligibility gate as far as Repository Service is concerned — it has no
  concept of "Level 1" or "Level 3" and doesn't need one.
- Level 3, when required, reuses the **existing** `Approval` table/workflow already used
  for requirements and architecture user-approval gates (`infrastructure/database/models.py::Approval`)
  — Repository Service's merge call is simply not issued by `engineering_head` until that
  `Approval` row is resolved. No new approval infrastructure anywhere in the platform.

**Critical governance rule (platform-wide, not just Engineering):** no agent may approve
an artifact it generated. Structurally enforced here because `code_review_lead`,
`engineering_head`, and the worker that authored a module are always three distinct
`agent_id`s in the hierarchy — this is asserted in code (§9 test requirement) and should
be adopted as a standing rule the same way for every future department (QA, Security,
DevOps, Docs) that introduces its own approval step.

### Decision 4 — `implementation_plan` is the Architecture→Engineering bridge

Jumping directly from architecture artifacts into per-agent tasks with no scheduling
artifact in between made dependency ordering (§3) implicit and untestable. Locked:
Engineering Head's first graph node (§8, `decompose`) does not just build an in-memory
task list — it **produces and registers `implementation_plan`** as a versioned Engineering
artifact (`artifact_type="implementation_plan"`, no user approval required — it is
mechanically derived, not a design decision) before any code generation starts. Ownership
sits with Engineering (not Architecture) because the plan is Engineering's own execution
schedule, built from Architecture's already-frozen outputs plus the dependency graph
Architecture already computes (`traceability_agent_worker` → `AgentContext.dependency_graph`,
see §4). Every Lead reads `implementation_plan`, never raw architecture artifacts directly,
for sequencing.

## 1. Department organization

The full 16-agent hierarchy is **already registered** in `core/runtime/factory.py::AGENT_REGISTRY`
(added ahead of implementation, same as Architecture's agents were pre-registered before M3.1).
No new agent IDs are introduced by this spec — implementation means giving each of these a
concrete `BaseAgent` subclass via `@AgentFactory.register(...)`, replacing the `GenericAgent`
placeholder each currently falls back to.

```
engineering_head                              (L3, parent: manager_agent)
  ├── backend_lead                            (L4)
  │     ├── api_implementation_worker         (L5)
  │     ├── database_layer_worker             (L5)
  │     ├── authentication_worker             (L5)
  │     └── business_logic_worker             (L5)
  ├── frontend_lead                           (L4)
  │     ├── component_worker                  (L5)
  │     ├── page_worker                       (L5)
  │     └── state_management_worker           (L5)
  ├── integration_lead                        (L4)
  │     ├── third_party_integration_worker    (L5)
  │     └── internal_integration_worker       (L5)
  └── code_review_lead                        (L4)
        ├── code_reviewer_worker              (L5)
        └── refactor_worker                   (L5)
```

**Escalation path:** worker → its L4 lead → `engineering_head` → `manager_agent`. A
worker never contacts `manager_agent` directly, and `engineering_head` never makes an
assumption on architectural ambiguity — it escalates to `manager_agent`, which may
re-open Architecture Review.

**Model tier:** `engineering_head` runs on `claude-opus-4-6` (matches `architecture_head`
and `security_head`). All L4/L5 agents default to `claude-sonnet-4-6`.

**Approval authority mapping (Decision 3):** `code_review_lead` = Level 1,
`engineering_head` = Level 2. Neither may hold Level 3 — that is always external
(`manager_agent` + a human), by design, so no single department can grant itself
production-level sign-off.

## 4. Artifact contracts

Every worker task is a `TaskInput` (per `core/runtime/context.py`) carrying an
`AgentContext` with `approved_artifacts` pre-populated — workers never query Postgres or
Qdrant directly.

**Per-worker input contract:**

| Worker | Required `approved_artifacts` keys |
|---|---|
| `database_layer_worker` | `implementation_plan`, `database_schema` |
| `authentication_worker` | `implementation_plan`, `security_architecture` |
| `business_logic_worker` | `implementation_plan`, `architecture_blueprint`, `database_schema` |
| `api_implementation_worker` | `implementation_plan`, `openapi_spec` |
| `component_worker` / `page_worker` | `implementation_plan`, `ui_blueprint` |
| `state_management_worker` | `implementation_plan`, `ui_blueprint`, `openapi_spec` |
| `third_party_integration_worker` / `internal_integration_worker` | `implementation_plan`, `integration_plan` |

**Per-worker output contract** — every worker returns an `AgentResult` whose `content` is
a `CodeModule`:

```python
class CodeModule(BaseModel):
    module_id:       str
    task_id:         str
    worker_agent_id: str
    files:           list[GeneratedFile]      # {path, content, language}
    artifact_type:   Literal["source_code"]
    depends_on:      list[str] = []           # other module_ids that must merge first
    test_files:      list[GeneratedFile] = [] # optional — worker may co-produce tests
    review_status:   Optional[str] = None     # None | "AI_APPROVED" | "AI_REJECTED" (Decision 3, Level 1)
    traces_to:       list[str] = []           # requirement_id / architecture_component ids (Engineering Coding Contract, §12)
```

Every `CodeModule` is registered in the Artifact Registry as `source_code` (versioned, no
user approval) before Code Review runs, so a rejected/refactored module produces a new
artifact version rather than mutating one in place.

**New artifact types registered by this spec** (extends the platform's artifact-type
registry from `AASC_Service_Specifications_v1.md`):

| Type | Produced by | Requires approval |
|---|---|---|
| `ui_blueprint` | architecture-service (Decision 1, Appendix A.1) | Yes (user) |
| `implementation_plan` | engineering-service (Decision 4) | No |

## 5. Coding standards (frozen once, enforced by `code_reviewer_worker`)

- **Python:** PEP 8, type hints on every function signature, `from __future__ import
  annotations`, Google-style docstrings on public classes/functions.
- **FastAPI:** one `APIRouter` per resource, `response_model` on every route,
  dependency-injected DB sessions — never a global session.
- **SQLAlchemy:** async ORM only (`AsyncSession`), `Mapped[...]` typed columns
  (matches `infrastructure/database/models.py` exactly), no raw SQL strings.
- **TypeScript:** strict mode, no `any`, functional components + hooks only, no class
  components.
- **Testing:** every backend module ships with `pytest` unit tests targeting ≥80%
  coverage (same threshold `coverage_analyzer_worker` in QA already enforces); every
  frontend component ships a smoke test.
- **Directory layout:** generated code lands in the monorepo layout Repository Service
  already scaffolds — `frontend/ backend/ infrastructure/ tests/ docs/` — engineering
  agents place files, they never invent new top-level directories.
- **Naming:** `snake_case` for Python modules/functions, `PascalCase` for
  classes/components, `SCREAMING_SNAKE_CASE` for constants.
- **Error handling:** every generated endpoint raises `HTTPException` with a specific
  status code, never a bare `except:`; every generated async function that calls
  Repository Service or an external API wraps the call and re-raises typed errors, not
  provider SDK exceptions.
- **Logging:** `structlog.get_logger(__name__)`, structured kwargs, never f-string
  interpolation into log messages.

`code_reviewer_worker`'s Level 1 checklist is exactly this list — nothing beyond it, so
review is deterministic and not a vague "does this look good" judgment call.

## 6. Repository interaction rules

This section supersedes the original engineering-service spec's "Engineering initializes
GitHub repository" language, and layers in the hierarchical PR model from Decision 2.

- Engineering agents **never** call `git` or a GitHub SDK. All repository operations are
  HTTP calls to Repository Service, or NATS publishes it already subscribes to.
- `engineering_head`, on receiving `architecture.design.completed`, calls
  `POST /repositories` (idempotent). Repository Service seeds the monorepo layout and
  `develop` branch — Engineering does not.
- Each worker calls `POST /branches` once per task
  (`feature/<task-id>-<slug>`), then `POST /commits` with its `CodeModule.files`
  translated into `FileChange` entries, carrying
  `CommitMetadata{project_id, workflow_id, task_id, agent_id, lead_id}`.
- Once every task in a Lead's current `implementation_plan` sprint has an `AI_APPROVED`
  `CodeModule` (Level 1, §7), that Lead calls `POST /branches` **once more** with
  `branch_type="integration"` (Appendix A.2), then re-commits each approved module's
  final file state onto that branch via `POST /commits`, then opens exactly one
  `POST /pull-requests` from the integration branch to `develop`.
- `engineering_head` issues `POST /pull-requests/approve` only after its own Level 2
  review (feature completeness, cross-team integration) — this is the call that actually
  flips Repository Service's `pull_requests.status` to `approved` and makes the PR
  merge-eligible.
- If Level 3 (Human Approval) is required for this PR (protected paths / production
  release / enterprise policy — flagged by `engineering_head` at Level 2 time),
  `POST /pull-requests/merge` is withheld until the corresponding `Approval` row
  (existing platform table) is resolved by a human via `manager_agent`'s normal
  approval-gate flow. Otherwise, merge follows immediately.
- Post-merge, worker feature branches that fed an integration branch are marked
  `merged` in Repository Service (not force-deleted) once the integration PR itself
  merges — full history stays inspectable.
- Failure contract (unchanged from the draft): `ProtectedBranchViolationError` /
  `PermissionDeniedError` from Repository Service are **not retryable** — escalate
  immediately. `ProviderUnavailableError` / `MergeConflictError` **are** retryable,
  capped at 3 attempts, then escalate. Matches
  `services/repository/workflows/repository_graph.py`'s existing non-retryable list.

**No new Engineering-owned Git state.** Engineering only needs its own task/module/
approval-level bookkeeping (§9) — Repository Service already owns `repositories`,
`branches`, `pull_requests`, `repository_events`.

## 7. Review cycle

```
Worker generates CodeModule
        ↓
Static Validation           syntax check, import check, lint (deterministic, no LLM)
        ↓
Architecture Compliance     does this module match the plan/schema/blueprint it was
                            scoped to? (code_reviewer_worker, checklist-driven)
        ↓
Code Review (Level 1)       §5 coding-standards checklist (code_reviewer_worker)
        ↓  fail
   Refactor Worker           applies reviewer feedback, re-emits CodeModule (new version)
        ↓  (loop back to Static Validation, max 3 cycles)
        ↓  pass → CodeModule.review_status = "AI_APPROVED"
Quality Check                final CritiqueResult, quality_score recorded on AgentResult
        ↓
(once all of a sprint's modules are AI_APPROVED)
Lead integration branch + single PR  (§6, Decision 2)
        ↓
engineering_head: Level 2 (Department Approval) → Repository Service approve_pull_request
        ↓
[optional] Level 3 Human Approval (Decision 3) → gates merge only
        ↓
Repository Service: merge
```

Reuses the existing `CritiqueResult` / `ReviewResult` dataclasses from
`core/runtime/context.py`. Cycle cap is 3, matching Architecture's `traceability_reruns`
cap and Repository Service's `max_retries` default.

**Separation of duties (Decision 3):** `code_reviewer_worker` must never review a module
where `worker_agent_id == code_reviewer_worker`'s own agent_id (structurally impossible
given the registry, but asserted in code and tested, §9). Same assertion applies to
`engineering_head` never being the Lead that authored the integration branch it is
granting Level 2 approval to — structurally true since `engineering_head` is a distinct
agent from every L4 lead.

**Exhaustion:** if a module fails Code Review 3 times, `code_review_lead` escalates to
`engineering_head`, which either reassigns with additional context or escalates further
to `manager_agent` for user input — it never force-merges a failing module.

## 10. Definition of done

Engineering Service is complete only when this full pipeline succeeds end-to-end,
against a real (or sandbox) GitHub repository via Repository Service:

```
Approved Requirements
        ↓
Approved Architecture (incl. ui_blueprint)
        ↓
Repository Service: create_repository (idempotent)
        ↓
implementation_plan generated & registered (Decision 4)
        ↓
Parallel Code Generation per sprint (Backend ∥ Frontend ∥ Integration)
        ↓
Review Cycle: Static → Architecture Compliance → Code Review (Lvl 1 AI_APPROVED) → Refactor loop, max 3
        ↓
Lead integration branch + single PR per team per sprint (Decision 2)
        ↓
engineering_head Level 2 Department Approval → Repository Service approve
        ↓
[optional] Level 3 Human Approval gates merge
        ↓
Repository Service: merge
        ↓
Sprint-gated loop until implementation_plan exhausted
        ↓
engineering.phase.completed
```

Additional exit criteria:

- [ ] All 16 agents have concrete `BaseAgent` subclasses (no `GenericAgent` fallback left)
- [ ] `code_reviewer_worker`'s checklist matches §5 exactly, nothing extra, nothing missing
- [ ] Zero direct Git/GitHub SDK calls anywhere under `services/engineering/`
- [ ] Appendix A amendments shipped and tested (`ui_blueprint` artifact, `integration`
      branch type) before any Engineering agent runs against Repository Service
- [ ] Separation-of-duties assertions (Decision 3) are unit-tested, not just documented
- [ ] W-Engineering graph compiles and its routing functions are unit-tested in isolation
- [ ] ≥120 tests passing, 0 regressions in the existing 148-test platform suite
- [ ] `docs/M3.3_Engineering_Service_Handover.md` written for M3.4 (QA Service) following
      the same handover format used for M3.2

## 11. Engineering Coding Contract

Every Engineering worker must produce code that is:

- **Buildable** — compiles/parses successfully (Static Validation, §7, before any
  human or LLM judgment is applied).
- **Runnable** — starts without manual fixes; no TODO-stub imports or placeholder
  connection strings left in generated files.
- **Testable** — includes or updates relevant tests (`CodeModule.test_files`, §4); a
  module with zero test coverage cannot reach `AI_APPROVED`.
- **Traceable** — `CodeModule.traces_to` links every module back to the requirement(s)
  and architecture component(s) it implements, extending the traceability discipline
  `traceability_agent_worker` already established in M3.1 into the code layer.
- **Reviewable** — passes the deterministic §5 checklist; nothing is approved on
  subjective judgment.
- **Idempotent** — re-running the same `EngineeringTask` does not duplicate or corrupt
  generated code (mirrors Repository Service's own idempotency guarantee on
  `create_repository`, §M3.2). Concretely: re-running a task overwrites that task's
  `CodeModule` files at the same paths and produces a new artifact version rather than
  appending duplicate code.

This is a universal quality contract — QA, Security, DevOps, and Docs can all assume
every module they receive already satisfies it, the same way Engineering can assume
every artifact it receives from Architecture is already frozen and approved.

## Appendix A — Required amendments to already-shipped services

Both are small and additive; neither reopens the M3.1 or M3.2 Definition of Done, they
extend it.

### A.1 — Architecture Service: add `ui_blueprint`

- New artifact type `ui_blueprint`, produced alongside the existing 9 architecture
  artifacts, subject to the same user-approval gate.
- Smallest implementation path: a new L5 worker (e.g. `ui_architect_worker`) under the
  existing `platform_design_lead` (which already runs 4 parallel workers and accepts
  partial success — this becomes a 5th), rather than a new L4 lead. Add
  `ui_architect_worker` to `AGENT_REGISTRY` the same way M3.1 added
  `security_architect_worker` etc.
- `architecture_reviewer_worker`'s cross-cutting consistency check gains one more rule:
  every page in `ui_blueprint` binds to at least one path in `openapi_spec`.

### A.2 — Repository Service: add `integration` branch type

- `services/repository/schemas/__init__.py::BranchType` gains a 4th member:
  `INTEGRATION = "integration"`.
- `services/repository/managers/__init__.py::build_branch_name()` gains a case for
  `branch_type == "integration"`: pattern `integration/<lead-id>-<slug>`.
- `infrastructure/database/models.py::Branch.branch_type` CHECK constraint extends from
  `('protected','feature','fix','hotfix')` to
  `('protected','feature','fix','hotfix','integration')` — a small additive Alembic
  migration (`0003_integration_branch_type.py`), not a rewrite of `0002`.
- No change to `merge_strategy` (still squash-only), no change to
  `PullRequestManager`, no change to `GitHubProvider` — integration branches are
  ordinary branches from Repository Service's point of view; only the naming policy and
  DB constraint need to know about the new type.

Both amendments are scoped tightly enough to land as fast-follow patches before
Engineering implementation starts, without reopening either service's frozen Definition
of Done.
